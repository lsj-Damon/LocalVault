# 本地离线桌面笔记软件需求文档（草案）

## 1. 项目概述

### 1.1 背景
需要一款运行在本机的桌面笔记软件，用于记录日常笔记，以及少量敏感信息，例如账号备注、API Key、证书片段、图片和图片说明。产品目标不是替代专业密码管理器，而是在“普通笔记软件一样好用”的前提下，提供必要的本地隐私保护能力。

### 1.2 产品定位
- 本地离线桌面软件
- 个人使用优先
- 笔记体验优先，隐私保护辅助
- Windows 优先，后续可评估跨平台

### 1.3 目标
- 支持快速记录、整理、搜索和回看笔记
- 支持在笔记中保存图片和少量敏感字段
- 所有数据仅保存在本机，不依赖云端
- 在不增加明显使用负担的前提下提供基础安全能力

### 1.4 非目标
- 不替代专业密码管理器
- 不做云同步
- 不做多人协作
- 不做复杂权限系统
- 不做企业级审计与合规能力

## 2. 目标用户与使用场景

### 2.1 目标用户
- 个人开发者
- 需要本地保存技术资料、截图、密钥备注的用户
- 希望离线可用且不想把内容放到云端的用户

### 2.2 核心场景
1. 记录普通笔记，例如命令片段、操作说明、待办说明。
2. 记录敏感备注，例如某服务的 Token 用途、测试账号信息、证书备注。
3. 保存截图、证书图片、界面参考图，并附加文字说明。
4. 按分类、标签或关键字快速检索历史记录。
5. 在换机或重装前导出并备份本地数据。

## 3. 产品原则

### 3.1 易用优先
- 像普通笔记软件一样直观，不引入复杂安全流程
- 常用操作不超过 2 到 3 步
- 编辑、搜索、切换笔记要足够快

### 3.2 本地优先
- 无网络依赖
- 核心功能在离线状态下完整可用
- 数据默认仅保存在本机

### 3.3 适度安全
- 对敏感内容提供必要保护
- 不要求每次查看都输入复杂口令
- 避免明文裸存关键敏感数据

## 4. 功能范围

### 4.1 MVP 功能

#### 4.1.1 笔记管理
- 新建、编辑、删除笔记
- 支持标题和正文
- 支持草稿自动保存
- 支持最近编辑排序

#### 4.1.2 组织方式
- 支持分类或笔记本
- 支持标签
- 支持收藏
- 支持归档

#### 4.1.3 搜索与筛选
- 支持按标题和正文全文搜索
- 支持按标签、分类、收藏状态筛选
- 支持最近打开或最近编辑列表

#### 4.1.4 内容类型
- 文本内容
- 基础格式化内容
- 图片附件或插入图片
- 图片说明和备注
- 敏感字段块，例如 Token、Key、账号备注

#### 4.1.5 隐私保护
- 软件启动主密码
- 闲置自动锁定
- 敏感字段默认隐藏，可手动显示
- 支持一键复制敏感字段

#### 4.1.6 数据管理
- 本地导出
- 本地导入
- 本地备份与恢复

### 4.2 v1 增强功能
- 多窗口或分栏查看
- 图片缩略图浏览
- 快捷键体系
- 系统托盘最小化与快速锁定

## 5. 详细功能要求

### 5.1 笔记编辑器
- 默认采用所见即所得的轻量编辑体验，不要求兼容 Markdown 编辑模式。
- 支持基础格式：标题、粗体、列表、代码块、分隔线。
- 支持粘贴图片或从本地选择图片插入。
- 图片支持填写标题、说明、来源备注。
- 自动保存频率可配置，默认开启。

### 5.2 敏感内容记录
- 在普通笔记中允许插入“敏感字段块”。
- 敏感字段块至少支持以下类型：密码、Token、API Key、自定义文本。
- 敏感字段默认遮罩显示。
- 支持点击显示/隐藏。
- 支持单独复制字段内容，不要求进入编辑态。
- 普通正文与敏感字段在存储层应区分处理，便于后续加密策略扩展。

### 5.3 笔记结构
每条笔记建议包含以下字段：
- `id`
- `title`
- `content`
- `category`
- `tags`
- `is_favorite`
- `is_archived`
- `attachments`
- `secret_fields`
- `created_at`
- `updated_at`

### 5.4 图片与附件
- 支持插入 PNG、JPG、WEBP 等常见格式。
- 图片可以内嵌显示，也可以作为附件展示。
- 图片元信息至少包含文件名、大小、添加时间、备注说明。
- 图片文件与笔记正文的关联关系需要稳定，避免移动文件后丢失。

### 5.5 搜索体验
- 搜索框全局可见。
- 输入关键字后快速返回标题、正文和标签匹配结果。
- 搜索需要覆盖敏感字段内容，但仅在应用已解锁状态下可用。
- 搜索结果中应能区分命中位置。
- 敏感字段命中时，默认只提示笔记与字段类型，不直接展示完整明文；点击后再解锁显示。

### 5.6 回收站与历史版本
- 删除的笔记先进入回收站，不直接永久删除。
- 回收站支持恢复与彻底删除。
- 笔记需要保留历史版本，至少支持按时间查看最近若干次变更。
- 历史版本应覆盖正文、标签、分类和敏感字段结构变化。
- 历史版本中的敏感字段仍需按当前安全策略处理。

### 5.7 导出格式
- 导出优先提供可读性更强的 `Markdown` 和 `HTML`。
- `HTML` 导出应尽量保留富文本样式、图片和基础结构。
- `Markdown` 导出允许对复杂富文本进行合理降级，例如自定义块转为代码块或注释块。
- 敏感字段导出时应提供选项：隐藏值导出、明文导出、完全排除。

## 6. 非功能需求

### 6.1 安全性
- 本地数据库或文件内容应加密存储，至少覆盖敏感字段和主数据文件。
- 主密码不明文保存。
- 自动锁定时间可配置，例如 5 分钟、15 分钟、30 分钟。
- 锁定状态下不能直接查看已打开的敏感内容。
- 敏感字段搜索仅在解锁会话中可用。

### 6.2 性能
- 冷启动时间目标：3 秒内进入可用状态。
- 常规搜索应在千级笔记规模下保持明显无卡顿。
- 单条笔记包含多张图片时仍需保证滚动和切换流畅。

### 6.3 可用性
- 界面尽量简单，主流程可自解释。
- 数据目录位置清晰，便于备份。
- 异常退出后自动恢复未保存内容。

### 6.4 可靠性
- 自动保存避免误丢数据。
- 导入导出要有明确的成功/失败反馈。
- 备份文件要能校验有效性。

## 7. 页面与信息架构

### 7.1 主界面
- 左侧：分类、标签、收藏、归档
- 中间：笔记列表
- 右侧：笔记编辑与预览区域

### 7.2 锁定界面
- 启动后先进入解锁页
- 锁定时仅显示应用名称、主密码输入框、解锁按钮

### 7.3 设置页
- 主密码修改
- 自动锁定时间
- 数据目录
- 导入导出
- 备份恢复

## 8. 数据设计建议

### 8.1 存储建议
- 元数据与正文可保存在本地数据库中
- 图片与附件保存在应用数据目录
- 数据库只保存索引、路径、摘要和关联关系

### 8.2 推荐方案
- MVP 优先考虑 `SQLite + 本地附件目录`
- 敏感字段内容单独加密存储
- 图片文件使用应用托管目录，不依赖原始路径
- 历史版本使用版本表或事件表保存

推荐该方案的原因：
- 本地桌面应用集成简单
- 便于做搜索、筛选和导出
- 后续可平滑扩展版本记录与索引

## 9. 技术方向建议

### 9.1 桌面技术候选
1. Electron
2. .NET WPF + CefSharp
3. .NET WPF / WinUI 原生富文本控件

### 9.2 推荐方向
在 `WebView2` 不可用的前提下，当前项目已确认采用 `Electron + React + TypeScript + TipTap + SQLite`。

原因：
- 不依赖系统 `Edge / WebView2`，运行环境更可控。
- 对富文本优先、图片、敏感字段块、`HTML/Markdown` 导出最友好。
- 你的参考项目里，`Joplin`、`Notesnook`、`TagSpaces`、`AFFiNE` 都证明了这条路线可行。
- 相比“WPF 壳 + 嵌入 Chromium 编辑器”，整套 UI 都在一套技术栈里，复杂度更低。

如果你仍然希望保留 `.NET` 桌面壳，次优方案是 `WPF + CefSharp + TipTap + SQLite`。

### 9.3 推荐技术栈
- 桌面壳：`Electron`
- 前端：`React + TypeScript + Vite`
- 编辑器：`TipTap`
- 状态管理：`Zustand`
- 路由：`React Router`
- 本地数据库：`SQLite + FTS5`
- 数据访问：`better-sqlite3` 或 `SQLite` 原生驱动
- 加密：`Argon2id + AES-256-GCM`
- 进程模型：`Main / Renderer / Preload`

### 9.3.1 保留 .NET 的备选方案
- 桌面壳：`.NET 8 + WPF`
- 嵌入浏览器：`CefSharp`
- 编辑器：`TipTap`
- 本地数据库：`SQLite + FTS5`
- 数据访问：`Dapper` 或 `Microsoft.Data.Sqlite`
- 加密：`Argon2id + AES-256-GCM`

说明：
- 该方案可以绕开 `WebView2`，但需要维护 `WPF + Chromium 编辑器` 两层 UI。
- 如果你非常明确要保留 `.NET`，可以走这条路。

### 9.4 推荐架构
- Renderer 层负责主窗口 UI、笔记列表、编辑器、设置、锁屏、回收站和历史版本面板。
- Main 层负责窗口、文件系统、数据库连接、导入导出、系统托盘和应用生命周期。
- Preload 层负责安全桥接，向前端暴露最小可用桌面能力。
- 本地数据层采用 `SQLite + 附件目录` 的混合存储结构。
- 正常正文与敏感字段都参与搜索，但敏感字段搜索结果默认掩码展示。

### 9.5 架构拆分建议
- `note_service`：笔记 CRUD、自动保存、归档、收藏
- `search_service`：标题、正文、标签、敏感字段搜索
- `secret_service`：主密码校验、密钥派生、敏感字段加解密
- `attachment_service`：图片导入、附件托管、缩略图缓存
- `history_service`：版本快照、版本回看、版本恢复
- `trash_service`：回收站、恢复、彻底删除
- `backup_service`：整库导出、导入校验、恢复

- 详细的项目目录结构、数据库表设计、搜索与加密策略见 `docs/plans/2026-04-24-local-secure-notes-implementation-architecture.md`

## 10. 边界与风险

### 10.1 产品边界
- 只提供“适度隐私保护”，不承诺达到专业密码库级别安全标准。
- 不将所有字段都按最高等级加密，否则会明显影响搜索和编辑体验。

### 10.2 主要风险
- “既要像普通笔记一样顺手，又要强安全”存在天然冲突。
- 图片、附件和敏感字段混合存储时，导入导出复杂度会增加。
- 如果全文都强加密，搜索能力会明显受限。

## 11. MVP 验收标准
- 用户可以创建、编辑、删除笔记。
- 用户可以为笔记添加分类、标签、收藏状态。
- 用户可以插入图片并写图片说明。
- 用户可以插入至少一种敏感字段块并进行隐藏/显示/复制。
- 软件在断网状态下完整可用。
- 软件支持主密码解锁与自动锁定。
- 用户可以导出和恢复本地数据。

## 12. 待确认事项
- 敏感字段搜索索引采用“解锁后内存索引”还是“数据库加密索引”。
- `Markdown` 导出对复杂富文本块的降级规则。
- 历史版本保留策略，例如按条数还是按时间保留。
